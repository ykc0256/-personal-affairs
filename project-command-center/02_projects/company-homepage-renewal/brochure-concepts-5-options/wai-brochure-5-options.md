# WAI 브로슈어 5가지 방향안

## 공통 전제

WAI는 완성된 SaaS나 운영 AI 제품이 아니라, 수처리 시설의 설계·시공·운영 데이터를 생애주기 관점으로 연결하는 `WATER AI 개념`으로 설명한다.

공통 기준:

- 외부 공개용 브로슈어
- 개발 중인 개념이므로 완료형 표현 지양
- `GWD → AGWD → WAI` 흐름은 언급 가능
- AGWD는 연결 단계로만 다루고, 세부 AI 모듈명은 제외
- 오믈렛, 외부 협력사, 수행 체계는 제외
- TRL 5, TRL 8, 18개월 상용화, 100억 매출 등 과제 목표 수치는 제외

---

# 1안. WATER AI 생애주기 환류형

## 방향

현재까지 논의한 기준안이다. WAI를 수처리 시설의 설계·시공·운영·재설계 데이터를 연결하는 WATER AI 개념으로 설명한다.

## 추천 제목

`WAI, 수처리 시설의 생애주기를 연결하는 WATER AI`

## 한 줄 정의

설계, 시공, 운영 데이터를 생애주기 관점으로 연결하고 다시 설계 기준으로 환류시키는 WATER AI 개념

## 본문 블록

| 블록 | 문구 |
| --- | --- |
| 왜 WAI인가 | 수처리 시설의 성능은 운영 단계만이 아니라 초기 설계 데이터에서 시작됩니다. |
| WAI 개념 | WAI는 설계·시공·운영 데이터를 생애주기 흐름으로 연결하는 WATER AI 개념입니다. |
| 관계 | GWD는 설계 데이터 기반, AGWD는 지능형 설계 자동화, WAI는 생애주기 데이터 환류 개념입니다. |

## 레이아웃

```text
[상단] WAI 제목 + WATER AI 한 줄 정의
[중앙] 설계 → 시공 → 운영 → 재설계 환류 다이어그램
[하단] 데이터 구조화 / 연결 / 축적 / 환류
```

## 적합한 경우

- 가장 안정적인 외부 공개용 시안
- WAI를 개념 자료로 설명해야 할 때
- 개발 중이라는 톤을 유지해야 할 때

---

# 2안. 운영 AI 한계 / 설계 데이터 출발형

## 방향

수처리 AI가 운영 최적화에만 머물러서는 부족하다는 문제의식을 전면에 둔다. WAI의 필요성을 설계 데이터에서 출발하는 구조로 설명한다.

## 추천 제목

`수처리 AI는 설계 데이터에서 시작됩니다`

## 한 줄 정의

운영 데이터만이 아니라 설계 단계의 판단까지 연결하는 WATER AI 방향

## 본문 블록

| 블록 | 문구 |
| --- | --- |
| 문제의식 | 운영 데이터만으로는 시설의 장기 성능을 모두 설명하기 어렵습니다. |
| 출발점 | 공정 조건, 설비 구성, 배치, 물량, 비용 판단은 설계 단계에서 만들어집니다. |
| WAI 방향 | WAI는 설계 데이터와 운영 결과가 다시 만나는 환류 구조를 지향합니다. |

## 레이아웃

```text
[상단] 문제 제기형 제목
[좌측] 운영 AI 중심의 한계
[우측] 설계 데이터에서 시작하는 WATER AI
[하단] 설계 판단 → 운영 결과 → 설계 기준 개선
```

## 적합한 경우

- WAI가 왜 필요한지 설득해야 할 때
- 운영 AI와 차별되는 메시지가 필요할 때
- 홈페이지 Problem/Solution 섹션과 연결할 때

---

# 3안. GWD → AGWD → WAI 로드맵형

## 방향

GWD, AGWD, WAI의 단계 관계를 가장 명확하게 보여준다. WAI를 갑자기 등장한 개념이 아니라 GWD 기반에서 발전하는 장기 방향으로 설명한다.

## 추천 제목

`GWD에서 AGWD, 그리고 WAI로`

## 한 줄 정의

설계 실행 엔진에서 지능형 설계 자동화, 생애주기 WATER AI로 이어지는 발전 방향

## 본문 블록

| 블록 | 문구 |
| --- | --- |
| GWD | 수처리 설계 데이터를 구조화하는 기반 엔진입니다. |
| AGWD | GWD 기반 설계 자동화를 지능형 설계 자동화로 고도화하는 단계입니다. |
| WAI | 축적된 설계 데이터를 운영과 재설계까지 연결하는 WATER AI 개념입니다. |

## 레이아웃

```text
[상단] 단계형 제목
[중앙] GWD → AGWD → WAI 3단계 로드맵
[하단] 설계 데이터 → 운영 데이터 → 설계 기준 개선
```

## 적합한 경우

- GWD와 WAI의 관계가 헷갈릴 가능성이 있을 때
- 내부/외부 설명에서 로드맵이 중요할 때
- AGWD를 적절히 언급하고 싶을 때

---

# 4안. 데이터 자산 / 지식 축적형

## 방향

WAI를 개별 프로젝트 데이터를 조직의 설계 지식으로 축적하는 관점에서 설명한다. “플랫폼 확장”보다 “데이터 자산화”에 가깝다.

## 추천 제목

`프로젝트 경험을 다음 설계 기준으로`

## 한 줄 정의

설계와 운영 경험을 축적해 다음 프로젝트의 설계 기준으로 환류시키는 WATER AI 개념

## 본문 블록

| 블록 | 문구 |
| --- | --- |
| 데이터 자산 | 설계 조건과 운영 결과는 다음 프로젝트를 위한 데이터 자산이 될 수 있습니다. |
| 지식 축적 | 반복되는 프로젝트 경험은 설계 기준과 검토 기준으로 축적되어야 합니다. |
| WAI 방향 | WAI는 프로젝트 경험을 다시 설계 지식으로 되돌리는 환류 구조를 지향합니다. |

## 레이아웃

```text
[상단] 데이터 자산화 제목
[중앙] 프로젝트 데이터 누적 → 운영 결과 → 설계 기준 개선
[하단] 조직 지식 / 기준 개선 / 재설계 환류
```

## 적합한 경우

- WAI를 기술 제품보다 지식 체계로 설명하고 싶을 때
- 운영 데이터 환류의 가치를 강조하고 싶을 때
- 장기 비전이지만 과장 없이 표현해야 할 때

# 5안. WATER AI CYCLE 중심형

## 방향

다이어그램 중심의 시각적 안이다. WATER AI CYCLE을 크게 보여주고, 텍스트는 최소화한다.

## 추천 제목

`WATER AI CYCLE`

## 한 줄 정의

설계 데이터가 운영 경험을 거쳐 다시 설계 기준으로 돌아오는 생애주기 데이터 환류 구조

## 본문 블록

| 블록 | 문구 |
| --- | --- |
| 구조 | 설계, 시공, 운영, 재설계가 하나의 데이터 흐름으로 연결됩니다. |
| 중심 | WAI의 핵심은 운영 결과가 다시 설계 기준으로 돌아오는 환류입니다. |
| 방향 | 완성형 제품이 아니라 WATER AI를 향한 생애주기 데이터 개념입니다. |

## 레이아웃

```text
[상단] WATER AI CYCLE 제목
[중앙] 대형 원형 순환 다이어그램
[주변] 설계 / 시공 / 운영 / 재설계 / 기준 개선
[하단] GWD → AGWD → WAI 작은 단계 도식
```

## 적합한 경우

- 이미지 시안에서 가장 직관적이고 시각적인 결과를 보고 싶을 때
- 글보다 개념도를 중심으로 보여주고 싶을 때
- 브로슈어를 홈페이지 섹션 이미지로도 재사용하고 싶을 때
