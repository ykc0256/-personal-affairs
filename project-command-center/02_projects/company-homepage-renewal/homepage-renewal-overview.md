## 오늘 기준 정리

- 기준일은 `2026-05-27`로 본다.
- 5/13~5/15 일정 협의와 메일 회신은 완료 상태다.
- 현재 남은 핵심은 5/26~5/28 기술소개자료/브로슈어 작성과, 5/29 이후 홈페이지 리뉴얼 본격 착수다.
- 브로슈어는 GWD와 WAI를 각각 1장짜리 자료로 분리 작성하는 방향으로 정리했다.
- 브로슈어 작성 기준 자료는 기존 홈페이지 시나리오보다 과제 발표자료와 과제 제안서를 우선한다.
- WAI 브로슈어는 외부 공개용 WATER AI 개념 자료로 작성한다. 흐름은 `GWD → AGWD → WAI`로 명시 가능하되, 오믈렛·세부 AI 모듈명·협력사 수행 체계는 제외한다.
- GWD/WAI 브로슈어는 단순 다이어그램 포스터가 되지 않도록 2~3개의 짧은 본문 설명 블록을 포함하되, 각 블록은 1~2줄 수준으로 줄인다.
- 이미지 시안 검토를 위해 GWD/WAI 전용 이미지 생성 프롬프트를 별도 MD 파일로 분리했다.
- 일정 구조 자체는 변경 없이 유지하고, 세부 확인 질문만 기술소개자료 이후 미팅에서 확정한다.

## 현재 결정된 방향

- 홈페이지 리뉴얼은 기술소개자료(브로셔)를 기반으로 제작한다.
- 기술소개자료/브로슈어는 `GWD 1장`과 `WAI 1장`을 분리하는 방향으로 검토한다.
- GWD 브로슈어는 현재 보유한 설계 실행 엔진, WAI 브로슈어는 WATER AI 원론과 설계·시공·운영·재설계 데이터 환류 구조를 중심으로 작성한다. 관계는 `GWD → AGWD → WAI`로 정리한다.
- GWD 본문 블록은 `왜 GWD인가`, `GWD 개념`, `엔지니어링 가치`로 구성한다.
- WAI 본문 블록은 `왜 WAI인가`, `WAI 개념`, `GWD / AGWD / WAI 관계`로 구성한다.
- 브로슈어의 최우선 참고자료는 과제 발표자료와 과제 제안서다.
- 기술소개자료는 최윤규 리더가 작성한다 (~5/28).
- 강수진 리더는 기술소개자료 및 홈페이지 디자인을 지원한다.
- 기술소개자료 작성 이후 이옥주 리더님과 미팅을 진행한다.
- 5/29부터 홈페이지 리뉴얼 본격 착수 (이옥주 리더 확인 완료).
- 오늘 기준으로 일정 구조는 유지하고 있으며, 세부 콘텐츠 확정은 기술소개자료 작성 이후 진행한다.
- 디자인은 홍보팀의 도움을 받아 구체화한다.
- TDG 브로셔는 홈페이지 구성과 메시지 표현의 참고 자료로 활용한다.
- 온라인 활용 콘텐츠(영상 등) 방향은 이옥주 팀과 별도 협의가 필요하다 (이옥주 리더 요청, 2026-05-15).
- WAI Planning 기본 구조에 대한 상세 논의가 필요하다.

> 총 3페이지 구성 (2026-05-14 최윤규 리더 초안, 이옥주 리더 확인)

| 페이지 | 주제 | 핵심 내용 | 참고 자료 |

| 2Page | WATER AI 환류구조 | GWD 설계 데이터 → 시공 → 운영 → 재설계 순환 구조 | `public/assets/reference/page_6.png` |
| 3Page | GWD 실제 활용 흐름 | 공정 조건 입력 → 계산 → 3D Layout → 도면/물량/비용 산출 → 결과 검토 | `public/assets/reference/page_6.png` |

**1Page 상세 방향**
- 기존 방식: 공정 계산·3D 설계·도면·물량·비용 산출이 각각 분리
- GWD 방식: 공정 조건 → 3D Layout → 도면/물량/CAPEX·OPEX 산출 일괄 연결
- 핵심 메시지: 반복 수정 감소, 산출물 일관성 향상, 설계 검토 속도 개선
- ※ GWD가 단순 AI 컨셉이 아닌 실제 설계 자동화 엔진임을 강조
- ※ 과제 발표자료 참고하여 수정 보완 예정

**2Page 상세 방향**
- 설계 데이터 → 시공 정보 → 운영 데이터 → 재설계 기준 개선 흐름
- WATER AI = 완성형 제품이 아닌 GWD 기반 데이터 확장 플랫폼 방향으로 정리
- WATER AI CYCLE 순환 구조 다이어그램 필요
- 개발 금액·개발시간·특허 내용 추가 예정

**3Page 상세 방향**
- 영상 업로드 또는 기존 프로그램 화면이 들어갈 구역 필요
- WAI Planning에서 사용자가 확인할 수 있는 것 위주로 정리
- ※ 기능 나열 아닌 실제 사용 흐름 중심
- ※ WAI Planning 기본 구조 상세 논의 필요
