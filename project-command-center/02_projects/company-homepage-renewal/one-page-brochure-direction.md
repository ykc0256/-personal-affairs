## 작성 목적

GWD와 WAI를 각각 1장짜리 브로슈어로 분리해 작성한다.

## 과제자료에서 가져와야 할 중심 논리

과제 발표자료와 제안서의 핵심 논리는 다음 순서다.

1. 글로벌 물산업 AI는 운영 단계에 집중되어 있으나, 설계 단계에는 큰 공백이 있다.
2. 수처리 설계는 공정 선택, 용량 산정, 3D 배치, 도면, BOM, CAPEX/OPEX가 분리되어 반복 수정과 오류가 발생한다.

4. GWD는 시뮬레이션과 실제 설계 산출물을 연결하는 수처리 설계 자동화 기반이다.
5. WAI / WaterAI는 설계 데이터를 시공, 운영, 재설계로 환류시키는 생애주기 데이터 지능화 방향이다.

# 2. GWD 1장 브로슈어 방향

## 브로슈어 역할

GWD 브로슈어는 “AI 비전”을 먼저 설명하는 자료가 아니라, 부강테크가 이미 확보한 **수처리 설계 실행 엔진**을 설명하는 자료다.

과제 제안서 기준으로 GWD는 시뮬레이션 결과와 실제 설계 산출물을 연결하는 미들웨어 성격의 기술이다. 따라서 GWD 브로슈어에서는 `설계 단계의 공백`, `기존 설계 방식의 비효율`, `GWD가 연결하는 산출물`, `검증 근거`를 한 장 안에서 압축해 보여준다.

## 추천 제목 후보

1. `GWD 기반 하·폐수 처리장 설계 자동화`
2. `공정 계산부터 설계 산출물까지 연결하는 GWD`
3. `수처리 설계의 반복 작업을 줄이는 설계 실행 엔진`
4. `Generative WWTP Design, 수처리 설계 자동화의 기반 엔진`

## 대표 설명 문장

> GWD는 수처리 공정 조건과 설계 로직을 연결해 Mass Balance, 3D Layout, 도면, BOM, CAPEX/OPEX 산출을 하나의 설계 흐름으로 구성하는 설계 실행 엔진입니다.

## 과제자료 기반 핵심 메시지

### Why Now

- 글로벌 물산업 AI는 운영 최적화, 센서 예측, 펌프 운영 등에 집중되어 왔다.
- 하지만 공정 선택, 설비 구성, 용량 산정, 3D 배치가 결정되는 설계 단계는 아직 AI와 자동화 적용이 지연된 영역이다.
- 설계 단계의 결정은 시설의 20~30년 운영비, 에너지 사용, 유지관리 구조에 영향을 준다.
- 따라서 수처리 산업의 AX 전환은 운영 단계뿐 아니라 설계 단계에서 시작되어야 한다.

### Problem

- 기존 수처리 설계는 공정 계산, Mass Balance, 3D 설계, 도면, BOM, 비용 산출이 분리되어 있다.
- 하나의 설계 조건이 변경되면 공정 재검토, Mass Balance 수정, 도면 재작성, 배관 수정, BOM 수정이 반복된다.
- 결과적으로 일정 지연, 오류 발생, 산출물 간 불일치, 엔지니어 개인 경험 의존 문제가 발생한다.
- GPS-X, BioWin 등 시뮬레이션 프로그램은 성능 검증에는 유용하지만, 실제 설계 산출물로 자동 전환되지는 않는다.

### Solution

- GWD는 수처리 공정 계산과 실제 설계 산출물을 연결한다.
- 공정 조건 입력 이후 물질수지, 구조물/기계/전기 규격, 3D Layout, 도면, 물량, CAPEX/OPEX 산출로 이어지는 흐름을 만든다.
- 설계 변경 시 후속 산출물의 일관된 갱신을 지원해 반복 작업과 검토 부담을 줄인다.
- 단순 3D 시각화나 도면 작성 도구가 아니라, 수처리 도메인 지식과 설계 로직이 내재화된 엔지니어링 엔진으로 설명한다.

## 1장 구성안

### A. 상단 Hero 영역

**제목**

GWD 기반 하·폐수 처리장 설계 자동화

**보조 문장**

공정 조건 입력부터 Mass Balance, 3D Layout, 도면, BOM, CAPEX/OPEX 산출까지 연결하는 수처리 설계 실행 엔진입니다.

**짧은 카피 후보**

- 수처리 설계의 AI 전환은 설계 데이터를 구조화하는 것에서 시작됩니다.
- GWD는 시뮬레이션과 실제 설계 산출물 사이의 간극을 연결합니다.
- 공정 계산에서 비용 검토까지, 분리된 설계 업무를 하나의 흐름으로 연결합니다.

### B. Problem / Solution 비교 영역

비교표만 두면 브로슈어가 포스터처럼 보일 수 있으므로, 짧은 문제의식 문장 1개를 함께 배치한다.

**본문 블록 — 왜 GWD인가**

> 하나의 설계 조건 변경이 공정 계산, 3D, 도면, BOM, 비용 산출까지 연쇄적으로 영향을 줍니다.

| 기존 설계 방식 | GWD 방식 |
| --- | --- |
| 공정 계산, 3D, 도면, BOM, 비용 산출이 개별 작업으로 분리 | 공정 조건을 기준으로 설계 흐름을 연결 |
| 설계 변경 시 각 산출물을 반복 수정 | 변경 조건을 후속 산출물에 일관되게 반영 |
| Mass Balance, 도면, 물량 간 불일치 가능성 | 공정 계산 결과가 3D, 도면, 물량 산출로 이어짐 |
| 엔지니어 경험과 수작업 의존도 높음 | 표준화된 설계 로직과 자동화 흐름 기반 |
| 설계 검토와 재작업에 긴 시간 소요 | 반복 수정 감소와 검토 속도 개선 지향 |

### C. GWD 핵심 흐름

핵심 흐름 다이어그램 주변에는 GWD의 성격을 짧게 설명한다.

**본문 블록 — GWD 개념**

> GWD는 공정 조건에서 도면, 물량, 비용 산출까지 이어지는 수처리 설계 실행 엔진입니다.

```text
프로젝트 생성
→ 공정계획 / 물질수지 / 공정용량 계산
→ 구조물·기계·전기 규격 산정
→ 3D Layout / 배관 / 전기 트레이 배치
→ PFD / P&ID / 평면도 / 단면도 / ISO 도면
→ BOM / 물량 / CAPEX·OPEX 산출
```

### D. 핵심 기능 카드

| 기능 | 설명 |
| --- | --- |
| 공정 계산 자동화 | 유입 조건과 공정 조건을 기반으로 물질수지와 공정 용량을 계산 |
| 설비·구조물 규격 산정 | 공정 계산 결과를 구조물, 기계, 전기 설비 규격으로 연결 |
| 3D Layout 생성 | 구조물, 설비, 배관, 전기 트레이 배치를 3D 기반으로 검토 |
| 설계 도면 생성 | PFD, P&ID, 평면도, 단면도, ISO, 전기 도면 등 산출물 생성 기반 |
| 물량·비용 산출 | BOM, 물량, CAPEX/OPEX 검토를 설계 흐름 안에서 연결 |

### E. Proof Points

# 3. WAI 1장 브로슈어 방향

## 브로슈어 역할

WAI 브로슈어는 과제 세부 기능이나 AGWD 모듈을 설명하는 자료가 아니라, 수처리 산업에서 왜 설계 단계의 데이터 구조화가 중요한지 설명하는 **외부 공개용 WATER AI 개념 자료**다.

여기서 WAI는 운영 단계의 AI만으로는 수처리 시설의 성능과 효율을 충분히 개선하기 어렵고, 설계 단계에서 만들어지는 공정 조건, 설비 구성, 배치, 물량, 비용 데이터가 시공과 운영을 거쳐 다시 설계 기준으로 돌아와야 한다는 논리를 설명한다. 독자가 한 장 안에서 “WAI는 완성된 플랫폼 소개가 아니라, 설계에서 운영까지 이어지는 수처리 데이터 지능화 방향”이라는 점을 이해하도록 구성한다.

GWD, AGWD, WAI의 단계 관계는 명시해도 된다. 다만 오믈렛, 세부 AI 모듈명, 협력사 수행 체계는 외부 공개용 WAI 브로슈어의 전면에서 제외한다.

## 추천 제목 후보

1. `WAI, 수처리 시설의 생애주기를 연결하는 WATER AI`
2. `WAI, 설계에서 운영까지 이어지는 수처리 데이터 지능화`
3. `설계·시공·운영 데이터를 환류하는 WATER AI`
4. `수처리 시설의 전 생애주기 데이터를 연결하는 WAI`

## 대표 설명 문장

> WAI는 수처리 시설의 설계, 시공, 운영 과정에서 발생하는 데이터를 생애주기 관점으로 연결하고, 축적된 데이터를 다시 설계 기준과 운영 개선으로 환류시키는 WATER AI 개념입니다.

## 핵심 메시지

### Why WAI

- 수처리 시설의 성능과 운영 효율은 운영 단계에서만 결정되지 않는다.
- 설계 단계의 공정 조건, 용량 산정, 설비 구성, 배치, 물량, 비용 판단이 장기 운영 구조에 영향을 준다.
- 하지만 현재 설계 데이터, 시공 데이터, 운영 데이터는 서로 단절되어 있어 현장 경험이 다음 설계 기준으로 돌아가기 어렵다.
- WAI는 이 단절을 줄이고, 설계·시공·운영·재설계가 하나의 데이터 흐름 안에서 연결되는 구조를 지향한다.

### WaterAI 환류 구조

- 설계 단계에서 공정 조건, 설비 구성, 3D, 도면, 물량, 비용 데이터가 구조화된다.
- 조달과 시공 단계에서 설계 데이터가 자재, 배치, 시공 정보와 연결된다.
- 운영 단계에서 유입수, 수질, 에너지, 성능 데이터가 축적된다.
- 운영 데이터는 재설계와 설계 기준 개선으로 환류된다.
- 이 순환 구조가 WATER AI의 핵심 개념이며, 장기적인 데이터 기반 설계·운영 체계의 기반이 된다.

GWD, AGWD, WAI의 관계는 큰 도식보다 작은 연결 문장 또는 짧은 단계 도식으로 처리한다. AGWD는 언급 가능하지만, 세부 모듈 설명은 넣지 않는다.

**본문 블록 — GWD / AGWD / WAI 관계**

> GWD는 설계 데이터 기반, AGWD는 지능형 설계 자동화, WAI는 생애주기 데이터 환류 개념입니다.

```text
설계 Engineering
  GWD로 설계 데이터 구조화
        ↓
조달·시공 Procurement & Construction
  자재, 배치, 시공 정보와 연결
        ↓
운영 Operation
  수질, 에너지, 성능, 유지관리 데이터 축적
        ↓
재설계 Re-Design
  운영 데이터를 설계 기준으로 환류
        ↺
다음 프로젝트의 설계 품질 개선
```

### 단계 관계형

> GWD가 수처리 설계 데이터를 구조화하는 기반 엔진이라면, AGWD는 이를 지능형 설계 자동화 과제로 고도화하는 단계입니다. WAI는 여기서 축적된 설계 데이터를 시공과 운영, 재설계까지 연결하는 WATER AI 개념입니다.

# 4. 두 브로슈어의 관계

WAI 브로슈어 하단:

> WAI의 출발점은 GWD 설계 실행 엔진이며, AGWD는 이를 지능형 설계 자동화 단계로 고도화하는 연결 단계입니다.

| 순서 | 자료 | 독자가 이해해야 할 것 |

| 2 | AGWD | GWD 기반 설계 자동화를 지능형 설계 자동화 과제로 고도화하는 단계다 |
| 3 | WAI | 이 기반에서 만들어진 설계 데이터를 시공·운영·재설계로 환류시키는 WATER AI 개념을 제시한다 |

2. GWD 브로슈어에서 현재 보유 기술과 검증 근거를 먼저 정리한다.
3. WAI 브로슈어에서는 AGWD를 연결 단계로 언급하되, 세부 모듈 설명은 줄이고 WATER AI의 환류 구조를 정리한다.
4. 두 브로슈어 모두 강한 목표 수치는 “목표”, “추진”, “지향”으로 구분해 표현한다.
5. 최종 문구는 홈페이지 리뉴얼에도 재사용할 수 있도록 과장 표현을 줄이고, 기술 기반과 확장 방향을 분리한다.

# 6. 검토 필요사항

## 확정 사항

- WAI는 `WATER AI 플랫폼`보다 `WATER AI 개념`으로 표현한다.
- GWD와 WAI의 관계는 명시한다. 정확한 흐름은 `GWD → AGWD → WAI`로 잡는다.
- GWD 공식 영문 풀네임은 `Generative WWTP Design`으로 사용한다.
- 브로슈어는 외부 공개용으로 작성한다.
- 공개 가능 수치는 `8년`, `50억+`, `특허`, 목표 수치 등은 사용 가능하되, `TRL 5`는 공개 브로슈어에서 제외한다.
- `세계 유일`, `설계 기간 80% 단축`, `설계 비용 50% 절감` 등 강한 표현과 목표 수치는 과제 발표자료 또는 제안서에 쓰인 표현이면 그대로 사용할 수 있다.
- WAI 브로슈어에서 AGWD는 언급 가능하다. 오믈렛, 세부 AI 모듈명, 협력사 수행 체계는 언급하지 않는다.

## 추가 확인

- 과제 발표자료 또는 제안서에 쓰인 강한 표현을 어느 위치에 배치할지 결정
- `설계 기간 80% 단축`, `설계 비용 50% 절감`을 GWD 브로슈어의 핵심 지표로 넣을지, 하단 기대효과로 넣을지 결정
- `세계 유일의 수처리 상세설계 미들웨어` 표현을 제목급 메시지로 쓸지, 본문 근거 문장으로만 쓸지 결정

## 이미지 / 시각자료 확인

- GWD 실제 화면 또는 3D Layout 캡처
- 공정 조건 입력 화면
- 도면 / BOM / CAPEX 산출 예시
- WATER AI CYCLE 다이어그램
- 설계·시공·운영·재설계 데이터 흐름 도식
- 운영 데이터가 설계 기준으로 환류되는 개념도
