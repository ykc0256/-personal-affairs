# Vendor 및 Item DB 설계 / 구조화

## 프로젝트 개요

| 항목 | 내용 |
| --- | --- |
| 목적 | Procurement 영역의 Vendor DB와 Item DB를 GWD에서 활용 가능한 구조로 반영하고, 향후 관리자 페이지 반영 기반을 만든다 |
| 담당 범위 | GWD 관점의 조회, 다운로드, 파일 저장, 평가 이력, 단가, Dashboard 요구사항 정리 및 Procurement팀 DB/코드 체계 반영 |
| 현재 상태 | 위험 |
| 우선순위 | 높음 |
| 기준일 | 2026-05-14 |
| 진행 로그 | `02_projects/vendor-item-db-progress.md` |

## 최근 확인 사항

2026-05-13 논의 기준으로, Procurement팀에서 이미 Vendor 관련 DB를 독자적으로 만든 것으로 보입니다. 따라서 이 프로젝트의 중심은 신규 DB를 처음부터 설계하는 것보다, Procurement팀이 만든 DB 구조와 코드 체계를 확인하고 GWD DB에 적재/반영하는 방향으로 재검토해야 합니다.

금주 또는 차주 중 Procurement팀 미팅이 예정되어 있으며, 해당 미팅에서 기존 DB 구조, 코드 정의, 데이터 전달 방식, GWD 반영 범위를 확인해야 합니다.

## 배경

`260120_개인사업계획서_최윤규.xlsx` 기준으로 Vendor DB Table 컬럼 설계 및 구조화는 1분기 계획, Item DB Table 컬럼 설계 및 구조화는 2분기 계획입니다. 2026-05-14 기준 실제 진행이 되지 않은 상태이므로 일정 리스크가 있습니다.

다만 Procurement팀에서 이미 DB를 별도로 구축한 것으로 확인되어, GWD 담당 업무는 `독자 설계`보다 `기존 Procurement DB/코드 체계의 GWD 반영`이 될 가능성이 높습니다.

## 목표

1. Procurement팀 기존 DB 구조와 코드 체계를 확보합니다.
2. GWD에 그대로 반영할 데이터와 재구조화가 필요한 데이터를 구분합니다.
3. Vendor DB, Item DB, Vendor-Item 관계 구조의 GWD 반영안을 정의합니다.
4. GWD 관리자 페이지에 필요한 조회, 다운로드, 파일 저장, 평가, 단가, Dashboard 요구사항을 정리합니다.
5. Procurement팀 요구사항 중 GWD에서 처리할 범위를 명확히 합니다.

## 현재 구조 판단

| 항목 | 판단 |
| --- | --- |
| 전체 방향 | 조회형 DB보다 업무 판단용 DB가 맞다. |
| 기자재 화면 | 분류 트리 + 목록 + 행 확장 패널 구조가 맞다. |
| 업체 화면 | 기자재 화면과 같은 분류 기준을 공유하고, 업체 평가와 보유 기자재를 행 확장 패널로 보여주는 구조가 맞다. |
| 단가 비교 | 별도 1차 메뉴로 두기보다 기자재 단가 이력에 흡수하는 구조가 맞다. |
| 단가 관리 | 단가 비교 페이지로 보내지 말고 기자재 확장 패널 내부에서 처리하는 구조가 맞다. |
| 단가 이력 | 기준일과 업체를 기준으로 설계가/실행가를 한 행에 묶는 구조가 맞다. |
| 상태 유지 | 기자재와 업체 화면 사이 이동 시 같은 분류 선택 상태를 유지하는 구조가 맞다. |

## 2026-05-22 화면 보완 반영

| 구분 | 반영 방향 |
| --- | --- |
| 기자재 분류 | 기계 / 배관 / 전기 및 하위 기자재 분류를 하나의 폴더형 트리로 표시한다. 하위 분류가 없으면 최상위 폴더만 보이고, 하위 분류가 추가되면 자동으로 접기/펼치기 대상이 된다. |
| 업체 화면 분류 | 업체 화면의 좌측 분류도 업체 분류가 아니라 기자재 분류 필터다. 따라서 전체 라벨은 `전체 기자재`로 표시한다. |
| 분류 상태 유지 | 사용자가 펼친 폴더는 다른 분류를 선택하거나 검색해도 유지되도록 한다. 기자재 화면에서 선택한 분류는 업체 화면으로 이동해도 동일하게 유지한다. |
| 분류 선택 UX | 별도 `선택` 버튼은 두지 않는다. 분류명 자체를 클릭하면 선택/필터가 적용되고, 선택된 분류 행은 배경색과 글자색으로 명확하게 표시한다. |
| 기자재 목록 | 취급업체수와 단위는 제거하고, 대표 업체·설계가·실행가·최고평가를 목록에서 바로 확인할 수 있게 한다. |
| 기자재 행 확장 | 기자재명 클릭 시 목록 아래 회색 패널에서 단가 변경 이력만 확인한다. 최신 단가는 별도 업체별 최신 정보 영역이 아니라 이력 테이블의 `최신` 상태로 표시한다. |
| 단가 변경 이력 | `기준일 + 업체`를 한 행 기준으로 두고, 설계가와 실행가를 같은 행에 나란히 표시한다. 각 가격의 전회 대비 변동액도 함께 표시한다. |
| 단가 관리 | 단가 비교 화면으로 이동하지 않는다. 현재 기자재 확장 패널 안에서 업체, 기준일, 설계가, 실행가, 출처, 비고를 입력하거나 오입력 정정을 처리한다. |
| 업체 목록 | 설계가와 실행가는 업체 목록에서 제외한다. 업체별 가격 조합은 기자재 상세/확장 패널에서 확인한다. |

## 1차 데이터 구조 가설

Procurement팀 기존 DB가 확인되기 전까지의 임시 구조입니다. 미팅 후 실제 구조에 맞춰 조정합니다.

### Vendor DB

| 컬럼 후보 | 설명 | 비고 |
| --- | --- | --- |
| vendor_id | 업체 ID | 기준 시스템 확인 필요 |
| procurement_vendor_code | Procurement팀 업체 코드 | 신규 확인 필요 |
| vendor_name | 업체명 | 조회 기준 |
| vendor_type | 업체 구분 | 물품, 공사, 용역, 기타 |
| dealer_name | 대리점명 | 별도 테이블 분리 여부 확인 필요 |
| performance_summary | 실적 | 기존 DB 보유 여부 확인 |
| financial_health | 재무건전성 | pass / non pass |
| user_rating | 사용자 평가 | 점수 체계 확인 필요 |
| revenue | 매출액 | 기준연도 필요 |
| capital_size | 자본금 규모 | 기준일 필요 |
| created_at | 등록일 | 시스템 컬럼 |
| updated_at | 수정일 | 시스템 컬럼 |

### Item DB

| 컬럼 후보 | 설명 | 비고 |
| --- | --- | --- |
| item_id | 아이템 ID | 내부 식별자 |
| procurement_item_code | Procurement팀 아이템 코드 | 신규 확인 필요 |
| item_code | GWD 아이템 코드 | 필요 시 매핑 |
| item_name | 아이템명 | 조회 기준 |
| category | 아이템 분류 | Dashboard/검색에 활용 |
| specification | 규격 | 단가 비교 기준 |
| unit | 단위 | EA, SET 등 |
| created_at | 등록일 | 시스템 컬럼 |
| updated_at | 수정일 | 시스템 컬럼 |

### Vendor Item DB

| 컬럼 후보 | 설명 | 비고 |
| --- | --- | --- |

| item_id | 아이템 ID | Item DB 참조 |
